# Оценка узких мест при миграции

## 1. Анализ ситуации

### 1.1 Контекст

Компания «Медикаменте» переходит от разрозненного контура (Excel, общий файловый сервер, 1С в
файловом режиме, ручные интеграции) к целевой архитектуре с единой медицинской системой (портал,
CRM, запись, платежи), API-интеграцией с лабораторией, защищенными каналами и аналитическим
контуром. Миграция сопряжена с ростом нагрузки (новые филиалы, мобильные клиенты) и необходимостью
сохранить непрерывность бизнес-процессов.

### 1.2 Ключевые процессы и компоненты, затрагиваемые миграцией

| Процесс                 | Текущие компоненты (As-Is)                      | Целевые компоненты (To-Be)                 | Риск для производительности                                              |
|:------------------------|:------------------------------------------------|:-------------------------------------------|:-------------------------------------------------------------------------|
| Запись и учет пациентов | Excel, общий диск, папки с файлами              | Единая медсистема, БД, объектное хранилище | Пики записи, конкуренция за ресурсы БД, медленный ввод при интеграции    |
| Оплата                  | Кассир, ККМ, 1С Бухгалтерия (OLE/TCP)           | Платежный контур, API к 1С/ККМ             | Задержки и таймауты интеграций, двойная запись в переходный период       |
| Лаборатория             | Реестры Excel, ручная передача                  | API + OAuth2, БД анализов                  | Внешняя зависимость, rate limits, очереди сообщений                      |
| Учет ТМЦ                | 1С Торговля и склад, обмен с бухгалтерией (OLE) | API-обмен, единый контур                   | Согласованность остатков и проводок, задержки синхронизации              |
| Доступ к медданным      | Файлы без формального аудита                    | RBAC/ABAC, шифрование, аудит               | Дополнительные проверки политик увеличивают latency при плохой настройке |
| Аналитика               | Excel, ручные выгрузки                          | Озеро данных, BI/ML                        | ETL/ингест, нагрузка на источники, размер данных                         |

## 3. Категории причин и возможные проблемы¬

### 3.1 Инфраструктура

- Недостаточная емкость вычислительных и сетевых ресурсов под пиковую нагрузку (запись, оплаты,
  мобильный доступ).
- Единая точка отказа или слабое резервирование БД и брокеров сообщений.
- Высокая задержка канала между филиалами и центральным контуром.

### 3.2 Программное обеспечение

- Избыточная сетевая связность микросервисов (чаттеринг), рост задержки из-за цепочек синхронных
  вызовов.
- Неоптимальные запросы к БД и отсутствие кэширования там, где ранее «держалось» в Excel локально.
- Утечки ресурсов и деградация под длительной нагрузкой без профилирования.

### 3.3 Интеграция

- Параллельная работа старого и нового контура (двойная запись) без четкого источника истины.
- Нестабильность или медленный ответ внешних API (лаборатория, платежи, 1С).
- Отсутствие идемпотентности и корректных политик повторов при сбоях интеграции.

### 3.4 Данные

- Неполная или «грязная» миграция: дубликаты пациентов, расхождение сумм и статусов.
- Большой объем файлов (сканы), долгий перенос и индексация.
- Несогласованность моделей данных между доменами (запись, оплата, медкарта).

### 3.5 Персонал

- Недостаток экспертизы в эксплуатации распределенной системы (Kubernetes, observability,
  интеграции).
- Ошибки пользователей в переходный период (гибридные процессы).

### 3.6 Процессы

- Отсутствие поэтапного плана cutover и отката (rollback).
- Недостаточное нагрузочное и интеграционное тестирование перед расширением на филиалы.
- Слабая регламентация инцидентов и эскалаций на этапе миграции.

### 3.7 Безопасность

- Каждый раз при запросе данных система проверяет, есть ли у сотрудника права. Если таких проверок много, ответ может замедляться.
- Шифрование данных требует дополнительных вычислительных мощностей. При массовом переносе старых файлов или выгрузке в аналитику это может стать заметной нагрузкой.
- Настройка защищённых каналов между сервисами требует времени на установку соединения. При большом количестве коротких запросов это даёт ощутимый оверхед.
- Если логировать каждое обращение к медицинским картам синхронно (сразу записывать в базу), это может тормозить основной процесс.
- Использование отдельного сервиса для хранения ключей шифрования (Vault) создаёт дополнительную сетевую зависимость. Если он недоступен, данные нельзя ни зашифровать, ни расшифровать.

## 4. Рекомендации и приоритеты

| Приоритет | Мера                                                                                                                                         | Ожидаемый эффект                                              |
|:----------|:---------------------------------------------------------------------------------------------------------------------------------------------|:--------------------------------------------------------------|
| **P0**    | Определить единый источник истины по доменам и сценарий миграции данных (поэтапный cutover, CDC или контролируемая двойная запись с сверкой) | Снижение расхождений и всплесков нагрузки на «латание» данных |
| **P0**    | Нагрузочное тестирование критических цепочек (запись, оплата, выдача результатов анализов) с целевыми SLO                                    | Выявление узких мест до продакшена                            |
| **P0**    | Внедрить observability: метрики, логи, трассировки, алерты по latency и ошибкам интеграций                                                   | Быстрая локализация деградации                                |
| **P1**    | Асинхронные очереди для интеграций с 1С, лабораторией, платежами; идемпотентные обработчики                                                  | Устойчивость к сбоям и пикам                                  |
| **P1**    | Кэш и выделенные read-модели для частых справочников и журналов                                                                              | Снижение нагрузки на БД                                       |
| **P2**    | Обучение персонала, runbooks, поэтапное включение филиалов                                                                                   | Снижение операционных ошибок                                  |

## 5. Выводы

Основные узкие места при миграции связаны не только с инфраструктурой, но и с **согласованностью
данных в переходный период**, **качеством интеграций** и **отсутствием измеримых SLO до запуска**.
Системное снижение рисков достигается связкой: четкая стратегия данных + нагрузочные испытания +
observability + устойчивые асинхронные интеграции.
